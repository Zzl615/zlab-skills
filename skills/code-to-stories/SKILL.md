# Code → User Story & Gherkin 生成器

将任意代码逆向转化为标准 User Story 和可直接使用的 Gherkin `.feature` 文件。

---

## 执行流程总览

```
代码输入 → [Step 1] 业务分析 → [Step 2] User Story → [Step 3] Gherkin → 结构化输出
```

完整输出包含五个章节，详见末尾「输出结构」。

---

## Step 1：代码分析

在生成任何文档前，必须完成以下分析。

### 1.1 识别代码边界

- **业务域**：判断代码所属领域（订单管理 / 用户认证 / 支付 / 库存等）
- **入口点**：API endpoint、方法签名、事件触发器
- **依赖关系**：外部服务调用、数据库操作、消息队列

### 1.2 代码模式 → 业务规则映射

| 代码模式 | 翻译为业务含义 |
|---|---|
| `if/else`、`switch` | 业务决策点、条件规则 |
| 数值边界（`> 0`、`<= 100`） | 业务约束条件 |
| `throw new Error` / 异常抛出 | 业务拒绝场景 |
| 数据库写操作（INSERT/UPDATE） | 业务状态变更 |
| 第三方服务调用 | 业务集成点 |
| 循环处理集合 | 批量业务操作 |
| 事务边界（`BEGIN/COMMIT`） | 原子性业务操作 |

### 1.3 识别角色（Actor）

从以下线索推断：
- 参数/变量命名（`adminUser`、`customerOrder`、`guestSession`）
- 权限校验逻辑（`hasRole()`、`isAuthenticated()`、`checkPermission()`）
- 业务上下文与领域术语

### 1.4 建立场景矩阵

识别所有执行路径：
- ✅ **Happy Path**：正常流程
- ❌ **Error Path**：异常/拒绝场景
- ⚠️ **Edge Cases**：边界条件、特殊输入

## Step 2：生成 User Story

### 格式模板

```
## US-[编号]: [简洁标题]

**Epic**: [所属功能模块]
**优先级**: [Must Have / Should Have / Could Have]

### 用户故事
作为 [角色]，
我希望 [完成某个操作]，
以便 [获得某种业务价值]。

### 背景说明
[2-3 句话说明该功能的业务背景]

### 验收标准（AC）
- AC1: [可测试的具体条件]
- AC2: [可测试的具体条件]

### 业务规则
- BR1: [从代码中提取的规则，可追溯到具体逻辑]
- BR2: ...

### 排除范围
- [明确说明不包含的内容]
```

### INVEST 质量检查

写完每个 Story 后，自检：
- **I**ndependent：故事间避免强依赖
- **N**egotiable：描述"做什么"而非"怎么做"
- **V**aluable：体现明确的业务价值
- **E**stimable：范围清晰，可被估算
- **S**mall：单个 Story 可在一个 Sprint 内完成
- **T**estable：验收标准必须可验证

## Step 3：生成 Gherkin Feature 文件

### 完整格式模板

```gherkin
# language: zh-CN

Feature: [功能名称]
  作为 [角色]
  我希望 [目标]
  以便 [价值]

  Background:
    Given [公共前置条件]
    And   [公共前置条件]

  # ─── 正向场景（Happy Path）──────────────────────

  @smoke @happy-path @US-[编号]
  Scenario: [正向场景标题]
    Given [系统初始状态]
    And   [额外前置条件]
    When  [用户执行的单一操作]
    Then  [可观测的预期结果]
    And   [附加断言]

  # ─── 参数化场景（边界条件）──────────────────────

  @boundary @US-[编号]
  Scenario Outline: [参数化场景标题]
    Given [使用 <参数> 的前置条件]
    When  [使用 <参数> 的操作]
    Then  [预期结果为 <期望值>]

    Examples:
      | 参数    | 期望值  |
      | 边界值1 | 结果1   |
      | 边界值2 | 结果2   |

  # ─── 异常场景（Error Path）──────────────────────

  @error-path @US-[编号]
  Scenario: [异常场景标题]
    Given [触发异常的前置状态]
    When  [导致错误的操作]
    Then  [系统返回的错误提示]
    And   [系统状态保持不变 / 事务回滚]
```

### 编写规范

**Given** — 系统初始状态，被动语态
- ✅ `Given 用户已登录且账户余额为 500 元`
- ❌ `Given 调用了 getUserBalance() 方法`

**When** — 用户的单一操作
- ✅ `When 用户提交金额为 200 元的转账申请`
- ❌ 多个 When（应拆分为独立 Scenario）

**Then** — 可观测的输出或状态变化
- ✅ `Then 系统显示"转账成功"并扣减余额`
- ❌ `Then userBalance 字段更新为 300`（实现细节）

### Tags 约定

| Tag | 用途 |
|---|---|
| `@smoke` | 冒烟测试必跑 |
| `@regression` | 回归测试 |
| `@happy-path` | 正向路径 |
| `@error-path` | 异常路径 |
| `@boundary` | 边界条件 |
| `@US-[编号]` | 关联 User Story |
| `@wip` | 开发中，暂不执行 |

## 输出结构

按以下五个章节顺序输出，不可省略：

### 🔎 章节一：业务逻辑分析摘要
识别到的核心业务域、主要角色、关键规则概述（约 200 字）。

### 📊 章节二：场景矩阵

| 场景编号 | 场景描述 | 路径类型 | 关联业务规则 |
|---|---|---|---|

### 📝 章节三：User Stories
按 Epic/功能模块分组，每个 Story 严格遵循 Step 2 格式。

### 🥒 章节四：Gherkin Feature Files
每个 Feature 独立代码块，标注 `gherkin` 语法，可直接保存为 `.feature` 文件。

### ⚠️ 章节五：分析局限性说明
因代码信息不足所做的假设、不确定点，标注需人工复核的部分。

---

## 核心约束

1. **可追溯性**：每条业务规则必须能映射回代码中的具体逻辑
2. **对应关系**：每个 AC（验收标准）必须有对应的 Gherkin Scenario
3. **业务语言**：不描述实现细节，只描述业务行为
4. **歧义处理**：代码存在歧义时，在章节五注明并给出最合理假设
5. **语言**：中文输出，技术术语保留英文
